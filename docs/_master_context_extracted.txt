2. Protocol Specification — UNMAPPED v0.2
2.1 Core Design Philosophy


2.2 Three Modules
Module 1 — Skills Signal Engine (SSE)  ✅ IMPLEMENTED
Converts heterogeneous, multi-source evidence into a standardized, portable Verifiable Skill Signal (VSS). Handles everything from formal diplomas to zero-credential community attestations.
Evidence input types: formal_credential, task_log, peer_attestation, community_navigator, mobile_payment, github_commit, assessment_score, observed_video
Taxonomy crosswalk: ISCO-08 (primary) → ESCO v1.2 (multilingual) → O*NET 28.1 (task detail) → local_overlay (country-specific informal extensions). NetworkX graph traversal.
Confidence scoring: Bayesian Beta conjugate update (no full MCMC for MVP). Tiers: Emerging (0–0.40) / Developing (0.40–0.60) / Established (0.60–0.80) / Expert (0.80–1.0)
Human layer rendering: Locale-aware. ISCO codes → plain-language titles. Confidence scores → 3-tier labels. Automation risk → plain-language trajectory string.
Portability outputs: JSON-LD + PDF profile card + QR code + SMS summary ≤ 160 chars + USSD pre-computation
Ownership controls: youth user can dispute any skill, add evidence, share profile, request deletion

Module 2 — AI Readiness & Displacement Risk Lens  ❌ NOT YET IMPLEMENTED
Given a VSS and locale context, produces an honest LMIC-calibrated readiness assessment showing how the skills landscape is shifting to 2035.
Required datasets: Frey-Osborne automation scores (base), ILO Future of Work task indices (LMIC calibration — critical), World Bank STEP microdata, Wittgenstein Centre 2025–2035 projections
Also uses: World Bank Human Capital Index, UNESCO Institute for Statistics, UN Population Projections, ITU Digital Development data
Output: per-occupation automation_risk_score, risk_trajectory_2035, durable_skills[], adjacent_skills[], resilience_score, confidence_interval. Plain-language for youth; full JSON for institutions.
LMIC calibration is mandatory: automation risk in Kampala ≠ Kuala Lumpur. Task-composition adjustment via ILO indices is required, not optional.

Module 3 — Opportunity Matching & Econometric Dashboard  ❌ NOT YET IMPLEMENTED
Bidirectional matching between VSS (youth) and Econometric Opportunity Signals (employer). Dual-view output: youth card (Amara) and aggregate dashboard (Kwame). All signals visible to user — none buried in algorithm.
Required econometric sources: ILO ILOSTAT (wages, employment growth), World Bank WDI, WBES (firm-level), Data360 Growth & Jobs, World Bank GLD, WBG Global Jobs Indicators
Also uses: B-READY, World Governance Indicators, Women Business & Law 2024, ILO Employment & Informality Stats
Mandatory display: ≥ 2 real econometric signals visible to youth user (wage range + sector employment growth)
Dual interface: youth view (honest, grounded, not aspirational matching) and policymaker view (aggregate signals, no PII)

The following must be configurable without changing a single line of code:

Language and script of the user interface
Automation exposure calibration (risk profiles differ by infrastructure context)

BONA network graph (accra_informal_2024.graphml vs yerevan_formal_2024.graphml)


2.6 Zero-Credential Path
When a user has no formal documents at all, the protocol does NOT fail. It triggers a Zero-Credential Flow:
Community Navigator Attestation Prompt — structured attestation request to registered navigator
Mobile Payment Signal Adapter — M-Pesa/MTN Mobile Money transaction history as business activity signal
Peer Attestation Micro-Survey — 3-question SMS survey to up to 5 named peers. Min 2 responses required.
Observed Task Log — short task video upload, transcribed by navigator
Profile generated with confidence_tier: 'Emerging' and explicit transparency statement

6. Critical Gaps, Risks & Design Decisions
6.1 The Biggest Missing Piece — Real Econometric Data

Current state: compute_wage_signal() and compute_growth_signal() in signals.py use hardcoded wage_bands and growth_channels from the country profile JSON. These are representative values, not fetched from ILOSTAT.
Required action: Either (a) pre-fetch and bundle the required ILOSTAT/WDI data as static JSON fixtures at build time (acceptable for hackathon — cite sources), or (b) implement an async data fetcher that calls the ILOSTAT API at parse time. Option (a) is strongly recommended for reliability.

6.2 The Biggest Missing Module — Automation Risk (Module 2)
Module 2 is the differentiating feature of UNMAPPED vs. a generic job board. Without it, the submission answers 'what can Amara do?' but not 'what will happen to her skills by 2035?' — which is the core of the brief's AI disruption requirement.
Minimum viable M2 for submission: bundle Frey-Osborne scores for the relevant ISCO codes, apply a simple LMIC adjustment factor from ILO Future of Work indices, and surface automation_risk_plain + automation_trajectory_2035 as two additional fields on the ProfileCard. This can be implemented as a new module in signals.py alongside the existing wage/growth signals.

6.3 Missing Policymaker View
The brief specifically requires a dual interface: one for the youth user, one for a policymaker viewing aggregate signals. Currently only the youth view exists. Without Kwame's dashboard, the submission does not fully satisfy the Module 3 requirement.
Minimum viable policymaker view: a read-only aggregate page (no individual PII) showing: total profiles by country/sector, average wage scores, automation risk distribution, NEET rate contextualisation. Can be added as a /dashboard route in the employer SPA (Phase 2).

6.4 spaCy Model Size — Docker Image Constraint
The xx_ent_wiki_sm model is pre-baked in the Docker image. Adding intfloat/multilingual-e5-small (~118 MB) + PyTorch CPU wheel significantly increases image size. The PROJECT_LOG notes a pending audit: confirm total image ≤ 2 GB. If using Railway (Step 17), Railway's free tier has a 512 MB RAM limit — E5-small at inference requires ~300–400 MB RAM. Set UNMAPPED_EMBED_DISABLE=1 for free-tier deployments.

6.5 LocalStorage PII Violation
The frontend uses localStorage to cache 'last input / locale / result' (mentioned in CHANGELOG alpha.1). The protocol spec Section 5.4 explicitly prohibits localStorage for PII on shared devices. The raw_input IS PII (it's Amara's personal story). This cache must be removed or scoped to non-PII data only (locale preference, last country_code) before production deployment.

6.6 Docker Compose Not Verified End-to-End
PROJECT_LOG LOG-0003 notes: 'docker compose up --build — NOT executed in this session — Docker Desktop daemon offline.' The full-stack compose has not been verified with a running Docker daemon. This must be the first test when setting up the dev environment.

6.7 BONA Not Implemented
The protocol spec defines BONA (Bidirectional Opaque Network Auditor) as a cross-cutting forensic layer for network capture detection, ghost listing identification, and programme leakage scoring. BONA is entirely absent from the codebase. It is a design concept in the spec only. For the hackathon demo, the network_entry field in ProfileCard is a simplified stub of BONA's youth-side Network Entry Map function.

6.8 API Contract Versioning — Stability Guarantee
The docs/api-contract.md states the POST /parse contract is 'stable' as of v0.3.1. However, Module 2 and Module 3 will require additional fields in the ProfileCard response (automation_risk_plain, automation_trajectory, opportunity_count, match_types). These must be added as optional fields only to avoid breaking the existing SPA. Never remove existing fields; only add new optional ones.

11. Brief Compliance Checklist

BROKEN SIGNALS | Education credentials don't translate into labour market signals. A secondary school certificate tells an employer almost nothing meaningful. Informal skills — phone repair, multilingual communication, mobile money operation — are entirely invisible to formal systems.

NO MATCHING INFRASTRUCTURE | Even where skills and jobs exist in the same place, the connective tissue is absent. Matching happens through informal networks that systematically exclude the most vulnerable youth.

UNMAPPED is a protocol, not a product. It is a thin, stateless transform layer between any Candidate CRM and Employer CRM. Every behaviour that varies by country — taxonomy, language, automation calibration, econometric sources, opportunity types, device constraints — is a pure config input. A single country_profile.json swap reconfigures the entire protocol for a new LMIC context.
Dimension | UNMAPPED Protocol Behaviour
Data ownership | Client CRMs own all primary data; protocol is stateless transform layer
Configurability | Pure config inputs — no code change for any country/language/sector swap
Vendor lock-in | None — Apache 2.0; any module replaceable; open JSON schemas
Offline capability | SMS/USSD fallback; compressed output; zero-bandwidth schema defined
Auditability | All scoring explainable via SHAP; human-readable provenance on every VSS
Shared-device PII isolation | No localStorage, sessionStorage, or device cache for PII. Profile retrieved via short code only.
